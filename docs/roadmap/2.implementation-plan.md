# 2. 实现规划

本文件用于把 `code-dance` 的第一阶段建设路径拆成可执行的实施计划。它不是总体路线图的替代品，而是对 [初始实现路线图](/home/zxr/work/github/code-dance/docs/roadmap/1.initial-plan.md) 的细化。

## 目标

第一阶段要完成一个可运行的 Rust 优先版本，具备以下能力：

- [x] 输入一个本地 Git 仓库路径
- [x] 识别 Rust workspace / crate 作为技术模块
- [x] 按周采样主线历史
- [x] 统计每个模块的 `loc`、`added`、`deleted`、`churn`
- [x] 将分析结果持久化到 SQLite
- [x] 通过本地 API 提供查询能力
- [x] 在 Web 页面展示时间序列图表

## 范围

### 本阶段包含

- [x] monorepo 工程骨架
- [x] 共享领域模型与契约层
- [x] Git 历史采样与文件树读取
- [x] Rust 模块探测
- [x] LOC 统计与增量指标计算
- [x] SQLite 存储
- [x] API 服务
- [x] Web 首屏可视化

### 本阶段暂不包含

- [ ] 多语言 provider 的完整覆盖
- [ ] 高精度语义级注释识别
- [ ] 分布式任务调度
- [ ] 多用户系统
- [ ] 高级权限管理

## 交付原则

- [x] 先打通主链路，再补横向能力
- [x] 先做 Rust provider，再扩展其他语言
- [x] Web 与未来 TUI 统一消费服务层接口
- [x] 每个阶段结束时都要有可验证产物

## Rust 优先实现策略

Rust 是第一阶段的主线，不应被放成一个普通子任务，而应贯穿分析链路设计。

建议按以下顺序落地：

1. [x] 先做 Rust fixture 思路与主链路
2. [x] 再实现 `RustWorkspaceProvider`
3. [x] 然后跑通 crate 级 LOC 快照
4. [x] 最后接入 API 与 Web 展示

原因：

- [x] crate 边界清晰，适合验证模块抽象
- [x] `Cargo.toml` 稳定，便于工程识别
- [x] 可以尽早暴露 provider 设计是否合理

## 风险与缓解

### Rust 仓库结构不统一

- [x] 先覆盖标准 workspace
- [ ] 独立 crate 作为第二优先级补充

### 大仓库分析耗时高

- [x] 第一阶段默认按周采样
- [ ] 结果落库，避免重复分析

### 模块归属歧义

- [ ] 统一 provider 优先级
- [ ] 支持手工配置覆盖自动识别

### 前后端契约漂移

- [x] 所有 DTO 放在 `packages/contracts`
- [x] 先定义契约，再落 API 与前端
