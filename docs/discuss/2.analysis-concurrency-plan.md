# 2. 分析链路并发实现方案讨论

## 背景

当前 `code-dance` 已经具备完整分析主链路，但并发能力还比较初级。

现状大致是：

- 单个 snapshot 内，`LOC` 统计已经有受限文件级并发
- sampled snapshot 外层仍按时间顺序串行处理
- `rust` / `node` analyzer 仍顺序执行
- API 层创建 analysis task 后立即启动，没有任务队列和全局并发预算

这意味着系统已经开始并发，但并发边界还没有统一设计。继续直接“加并发”很容易把 Git、磁盘、SQLite 和进度写入一起推成瓶颈。

## 讨论问题

需要明确三个问题：

1. 当前阶段最值得先做的并发层是哪一层
2. 哪些层不应该直接放开并发，而应该先补基础设施
3. 如何把“单任务内并发”和“多任务之间并发”统一到一个可控模型里

## 候选方案

### 方案 A：继续放大文件级并发

做法：

- 保持当前 snapshot 串行
- 保持 analyzer 串行
- 仅提高单个 snapshot 内文件读取并发度

优点：

- 改造最小
- 复用现有 `loc-counter`
- 对中等规模仓库会有直接收益

问题：

- 当前主要成本仍然是大量 Git 调用，而不是 JavaScript 计算
- 并发度升高后，收益会很快递减
- API 进度写入频率和 Git 子进程数会一起放大

结论：

- 这是保守优化，不是主方案
- 只适合作为基线能力和调参项存在

### 方案 B：直接做 snapshot 级并发

做法：

- 让多个 sampled snapshot 同时执行模块识别、LOC 统计和 diff 读取

优点：

- 理论收益最大
- 对 `per-commit`、高密度 `daily` 采样最有价值

问题：

- 当前结果生成过程仍带有顺序依赖
- 进度模型会从线性推进变成多任务聚合
- 会显著抬高 Git I/O 峰值和内存占用

结论：

- 这是后续最重要的结构性优化
- 但不适合在没有统一预算、缓存和进度节流前直接落地

### 方案 C：先做统一并发预算，再逐层开放并发

做法：

- 把并发问题拆成三个层次：
  - 任务级调度
  - 单任务内 analyzer 级并发
  - 单 analyzer 内 snapshot / 文件级并发
- 为整个 analysis task 建立统一预算，而不是每层自己开并发池
- 先把缓存、进度节流、结果顺序稳定性补齐，再放开更高层并发

优点：

- 能避免多层并发叠加
- 更适合后续扩展更多语言 analyzer
- 可以把“多任务吞吐”和“单任务延迟”统一纳入调度模型

问题：

- 实现会比单纯调大并发参数复杂
- 需要先整理 analyzer 主流程和 API task 管理

结论：

- 这是当前阶段最合理的主方案

## 推荐结论

推荐采用“统一预算、分阶段放开”的方案，优先级如下。

### Phase 1：收口当前并发模型

目标：

- 先把现有文件级并发变成稳定基础设施，而不是继续盲目提并发

建议实现：

- 保留 `packages/analyzer/src/languages/shared/loc-counter.ts` 的受限文件级并发
- 把并发度显式纳入 analyzer 配置，例如：
  - `fileReadConcurrency`
  - `snapshotConcurrency`
  - `analyzerConcurrency`
- 默认值保守：
  - `fileReadConcurrency = 6`
  - `snapshotConcurrency = 1`
  - `analyzerConcurrency = 1`
- 对进度回调做节流，避免每处理一个文件都持久化一次

验收标准：

- 默认配置下大仓库分析不会因进度写入过多而退化
- 取消任务时不会留下部分写入的残缺结果

### Phase 2：补缓存，不急着继续加并发

目标：

- 优先减少 Git 调用次数

建议实现：

- 在同一 revision 范围内引入文件内容读取缓存
- 优先设计 revision + filePath 维度的内存缓存
- 后续再评估是否引入批量 blob 读取，如 `git cat-file --batch`

原因：

- 当前分析链路更像 Git I/O 受限，而不是 CPU 受限
- 减少总工作量通常比增加更多并发更有效

验收标准：

- 同一 analysis task 内不会重复读取同一 revision 的同一文件内容
- 性能基准中 Git 调用次数有可观下降

### Phase 3：引入 analyzer 级并发

目标：

- 在混合仓库中同时执行 `rust` 和 `node` analyzer

建议实现：

- 修改 `packages/analyzer/src/history-analyzer.ts`
- 由当前顺序执行：
  - `for (const analyzer of analyzers) await analyzer.run(input)`
- 调整为受限并发执行
- analyzer 间共享同一个并发预算上下文，而不是各自独立扩张文件并发

关键约束：

- 最终 `snapshots` 必须完全一致才能合并
- 合并前必须继续校验 snapshot 时间线

验收标准：

- 混合仓库中总耗时下降
- 不会出现 `rust` / `node` 结果时间线不一致的合并错误

### Phase 4：引入 snapshot 级并发

目标：

- 提升高密度采样场景下的处理速度

建议实现：

- 每个 snapshot 独立产出原始结果：
  - 当前 revision 的模块集合
  - 当前 revision 的 `loc`
  - 与上一个 snapshot 的 diff 聚合结果
- 取消在 snapshot 处理中直接做“消失模块补零”
- 改为在全部 snapshot 完成后统一按时间顺序补齐、排序和合并

关键约束：

- 并发执行不等于结果无序输出
- 最终持久化前必须恢复稳定顺序

验收标准：

- `weekly` / `daily` / `per-commit` 结果顺序稳定
- 结果与串行模式在指标上完全一致

### Phase 5：API 层任务调度

目标：

- 不再让 `/api/analyses` 创建后立刻裸跑

建议实现：

- 在 `apps/api/src/server.ts` 中引入内存任务调度器
- 新建任务先进入队列，再由 scheduler 按预算启动
- 预算至少分两层：
  - 全局最大并发任务数
  - 同一 repository 最大并发任务数固定为 `1`

推荐默认值：

- `maxConcurrentAnalyses = 2`
- `maxConcurrentAnalysesPerRepository = 1`

原因：

- 同一仓库的多任务并发价值低，资源争用大
- 不同仓库允许少量并行，能提高整体吞吐

验收标准：

- 同仓库重复提交分析请求时不会同时压 Git
- 服务重启恢复任务时仍遵守调度规则

## 不推荐的实现方式

### 1. 每一层都自己维护并发池

例如：

- API 同时跑多个任务
- 每个任务同时跑多个 analyzer
- 每个 analyzer 同时跑多个 snapshot
- 每个 snapshot 再跑多个文件 worker

问题：

- 这是典型的并发乘法膨胀
- 最终很难知道系统真实并发度是多少

### 2. 过早引入 `worker_threads`

当前热点更偏向 Git I/O，不是纯 CPU。

如果底层仍然是大量 Git 子进程调用，那么把调度搬到 worker 并不会从根本上解决问题，反而会增加工程复杂度。

### 3. 先做任务级高并发，再补缓存和限流

这会最先把 API、Git 和 SQLite 推成瓶颈，风险最大。

## 推荐的最小落地顺序

1. 稳定文件级并发配置和进度节流
2. 引入 revision 文件读取缓存
3. 做 analyzer 级受限并发
4. 重构 snapshot 聚合模型，支持 snapshot 级并发
5. 在 API 层加入任务调度与按仓库限流

## 后续动作

- 在 analyzer 输入参数中补统一并发配置对象
- 在 API 层补最小 scheduler，而不是继续直接 `registerAnalysisTask`
- 为串行模式和并发模式建立结果一致性测试
- 增加大仓库性能基准，比较：
  - 无缓存 + 串行
  - 文件级并发
  - 文件级并发 + 缓存
  - analyzer 级并发
  - snapshot 级并发

## 最终结论

当前阶段不建议把“并发实现”理解为单纯把线程或 Promise 开得更多。

更合理的实现路径是：

- 先统一并发预算
- 先减少 Git 调用次数
- 再逐层开放 analyzer 和 snapshot 并发
- 最后让 API 层负责多任务调度

这样更符合 `code-dance` 当前以 Git I/O 为主瓶颈的现实，也更适合后续继续接入更多语言和更多前端入口。
