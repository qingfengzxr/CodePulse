# 模块演化图表设计

## 背景

当前 `code-dance` 已经能够生成模块级时间序列，并在 Web 页面展示第一版 crate `LOC` 折线图。

但随着模块数量增多，单纯把所有模块同时画成折线会迅速失去可读性，主要问题包括：

- 模块数量大时，图例和折线相互遮挡
- 折线图适合少量对象对比，不适合同时展示数十到数百个模块
- 不同分析问题需要不同图形表达，单一图表无法同时回答“总量变化”“结构占比”“热点活跃”“排名变化”等问题

因此需要明确一套更稳定的图表设计方案，指导后续 Web 展示层扩展。

## 目标

- 为不同分析问题选择更合适的图表类型
- 明确哪些图表适合作为主图，哪些只适合作为辅助视图
- 形成统一的信息架构，避免页面后续变成图表堆砌
- 为后续 `loc/added/deleted/churn` 多指标展示提供清晰设计边界

## 非目标

- 本文不定义具体前端组件实现细节
- 本文不定义具体 ECharts option 结构
- 本文不讨论 TUI 的可视化细节
- 本文不处理真实 diff 指标的计算实现，指标口径以现有设计和讨论结论为准

## 设计原则

### 1. 一个图表只回答一类核心问题

不要让同一个图同时承担“总量趋势”“模块对比”“活跃热点”“占比结构”四种任务。

### 2. 默认优先支持高密度数据可读性

面对几十到几百个模块时，默认展示必须先解决噪音问题，再谈细节交互。

### 3. 主图看全局，辅图看细节

主图负责帮助用户快速理解仓库结构和历史阶段变化；辅图负责回答具体模块、具体时间点、具体指标的问题。

### 4. 时间序列和快照视图分离

有些图适合看“变化过程”，有些图只适合看“某一时刻结构”。这两类视图不能混用。

## 分析问题与图表映射

建议把常见分析问题分成以下几类：

### 1. 仓库整体规模如何变化

推荐图表：

- 堆叠面积图
- 总量折线图

适合指标：

- `loc`

说明：

- 堆叠面积图适合看总规模增长，以及不同模块在总量中的占比变化
- 如果只看全仓总体规模，一条总量折线就够，不需要同时展示所有模块

### 2. 少量核心模块的趋势如何对比

推荐图表：

- 多模块折线图

适合指标：

- `loc`
- `added`
- `deleted`
- `churn`

说明：

- 折线图适合对比少量模块，不适合全量模块
- 默认应只显示 Top N 或用户手动选中的模块
- 模块很多时，应通过图例筛选、搜索、分组和范围切换减少噪音

### 3. 某个时间点谁最大、谁最活跃

推荐图表：

- 横向条形图

适合指标：

- `loc`
- `added`
- `deleted`
- `churn`

说明：

- 条形图是模块排行的主力图形
- 横向布局更适合长模块名
- 适合支持“当前快照排行”“某个时间窗口排行”“最近一次采样排行”

### 4. 哪些阶段最动荡，哪些模块在那个阶段最活跃

推荐图表：

- 热力图

适合指标：

- `added`
- `deleted`
- `churn`

说明：

- 横轴为时间，纵轴为模块，颜色表达指标强度
- 热力图不擅长精确读值，但非常适合扫出热点区间
- 对工程活动分析来说，它比全量折线图更有效

### 5. 模块排名如何随时间变化

推荐图表：

- Bump Chart

适合指标：

- `loc` 排名
- `churn` 排名

说明：

- 当用户关心“哪些模块正在上升、哪些模块逐渐边缘化”时，排名变化图非常直观
- 该图适合少量头部模块，不适合全部模块

### 6. 某一时刻的结构占比和层级关系如何

推荐图表：

- Treemap

适合指标：

- `loc`
- 某时间窗口内的 `churn`

说明：

- 当系统支持展示模块分组、workspace、子系统层级时，Treemap 比饼图更适合展示结构占比
- 它适合做快照结构图，不适合表达连续时间变化

### 7. 模块规模分布是否集中

推荐图表：

- 直方图

适合指标：

- 模块 `loc` 分布
- 模块 `churn` 分布

说明：

- 直方图适合回答“这个仓库是少数超大模块主导，还是长尾分散”
- 它适合做分析辅助图，不适合作为主图

## 不推荐作为主图的图表

### 饼图

不推荐原因：

- 模块一多就不可读
- 只适合展示单一时间点的占比
- 无法表达时间变化
- 长模块名和小扇区会严重影响可用性

结论：

- 不作为主图
- 如需展示某一时刻占比，优先使用 Treemap 或横向条形图

### 全量多折线图

不推荐原因：

- 当模块数量达到几十以上时，几乎无法阅读
- 图例管理困难
- 颜色编码失效
- Hover 和选中交互会变得混乱

结论：

- 不应作为默认展示方式
- 只适合少量模块对比

## 推荐的页面信息架构

对于 Web 分析详情页，建议采用“概览 + 聚焦 + 活跃度”的布局。

### 第一层：概览层

目标：

- 让用户快速理解整个仓库的规模和结构变化

建议内容：

- 仓库总 `LOC` 趋势
- 模块堆叠面积图
- 当前时间点 Top 模块摘要

### 第二层：聚焦层

目标：

- 让用户查看少量关键模块的细节趋势

建议内容：

- Top N / 手选模块折线图
- 指标切换：`loc / added / deleted / churn`
- 时间范围缩放
- 模块搜索和筛选

### 第三层：活跃度层

目标：

- 让用户发现重构期、热点期和高 churn 模块

建议内容：

- 时间 x 模块热力图
- 某时间窗口活跃模块排行
- 异常波动模块列表

### 第四层：结构层

目标：

- 让用户理解某个时间点或某个窗口内的模块结构

建议内容：

- 横向条形图
- Treemap
- 排名变化图

## 第一阶段推荐落地顺序

为了控制复杂度，建议分阶段推进，而不是一次性做全套图表。

### Phase 1

- 优化现有折线图默认展示
- 增加 Top N 与手动筛选能力
- 增加时间范围缩放

目标：

- 先让现有主图在多模块场景下可用

### Phase 2

- 增加模块堆叠面积图
- 增加某时间点横向条形图

目标：

- 补齐“结构变化”和“当前排行”两个高价值视图

### Phase 3

- 增加 `added/deleted/churn` 热力图
- 增加时间窗口排行

目标：

- 让系统开始具备工程活动分析能力

### Phase 4

- 增加 Bump Chart
- 增加 Treemap
- 增加技术视图与业务视图切换

目标：

- 支持更复杂的结构和排名分析

## 数据与交互要求

后续图表设计要成立，数据层和交互层至少需要满足以下条件：

### 数据侧

- 能按时间点查询模块级 `loc`
- 能按时间窗口查询模块级 `added/deleted/churn`
- 能返回模块分组信息，支持技术模块和展示模块映射
- 能支持 Top N 排序与聚合

### 交互侧

- 模块筛选
- 模块搜索
- 指标切换
- 时间范围缩放
- 技术视图 / 业务视图切换
- 噪音模块隐藏

## 风险与注意事项

### 1. 图表数量过多会损害可用性

不是图越多越好。页面必须围绕“用户想回答什么问题”来组织，而不是围绕“还能塞什么图”来组织。

### 2. 错误指标会让图表失真

如果 `added/deleted/churn` 继续使用净变化近似值，那么热力图、活跃排行、柱状图都可能误导用户。

### 3. 全量展示会快速失控

模块数量一旦增多，任何默认全量展开的方案都会很快失效，因此筛选和聚合不是可选项，而是基础能力。

### 4. 颜色不能承担主要区分责任

模块很多时，颜色只适合作为辅助编码，不能依赖颜色本身完成识别。必须结合排序、筛选、聚焦和交互。

## 当前结论

本次设计结论如下：

- 折线图保留，但只作为少量模块趋势对比图
- 堆叠面积图应成为总览主图
- 横向条形图应成为快照排行主图
- 热力图应成为 `added/deleted/churn` 活跃度主图
- Treemap 和 Bump Chart 适合作为进阶辅助视图
- 饼图不作为主图
- 直方图只作为分布分析辅助图

这意味着后续 Web 展示层应从“单一折线图页面”逐步演化为“概览、聚焦、活跃度、结构”四层视图体系。
