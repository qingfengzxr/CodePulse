# 分析链路并行化设计

> 版本：v0.1
> 更新时间：2026-04-09
> 变更摘要：新增分析链路并行化设计，明确当前并行维度、后续加速空间与推荐落地顺序

## 1. 背景

`code-dance` 当前已经具备完整的离线分析主链路：

- 枚举 Git 历史
- 按采样维度选取 snapshot
- 在每个 snapshot 上识别模块
- 统计模块 `loc`
- 读取相邻 snapshot 的 diff stat，推导 `added`、`deleted`、`churn`
- 落库并通过 API 提供查询

随着仓库规模上升，分析耗时会明显增长。影响耗时的主要因素包括：

- sampled commit 数量增多
- 单个 snapshot 下模块数量增多
- 模块覆盖文件数增多
- 底层 Git 子进程调用次数增多

因此需要单独定义并行化设计，回答以下问题：

- 当前已经并行化的是哪一层
- 后续还能从哪些维度继续加速
- 哪些优化值得优先做，哪些不应过早复杂化

## 2. 目标

- 说明当前分析链路中的实际并行维度
- 明确各层可并行化空间及其收益与成本
- 区分“增加并发”与“减少总工作量”两类优化
- 给出可逐步落地的实施顺序，避免一次性引入过多机制

## 3. 非目标

- 本文不定义具体前端进度 UI 表现
- 本文不引入分布式调度或多机执行模型
- 本文不讨论更高精度语义级分析
- 本文不直接定义某一种语言 provider 的模块规则细节

## 4. 当前执行模型

### 4.1 当前分析主流程

当前 analyzer 的主流程可以概括为：

1. 读取分支上的主线 commits
2. 按 `weekly`、`daily`、`monthly`、`per-commit` 等粒度采样
3. 逐个 snapshot 执行：
   - 识别当前 revision 的模块
   - 统计当前模块 `loc`
   - 读取与上一个 snapshot 之间的 `numstat`
   - 归因到模块并生成 `added`、`deleted`、`churn`
   - 输出本 snapshot 的 points
4. 所有 snapshot 完成后统一持久化结果

### 4.2 当前已经落地的并行维度

当前真正已经落地的并行，主要只有一层：

- 单个 snapshot 内，按文件读取 revision 内容时使用受限并发池

这意味着当前并行化主要发生在 `loc` 统计阶段，而不是整个分析流程。

### 4.3 当前仍然是串行的部分

以下层次目前仍然保持串行：

- sampled snapshot 外层仍按时间顺序逐个处理
- 单个分析任务中的不同语言 analyzer 仍顺序执行
- diff stat 归因阶段仍逐条串行处理
- API 侧对多个 analysis task 没有显式的全局并发调度策略

### 4.4 当前瓶颈的本质

当前系统的主要瓶颈更偏向 Git I/O，而不是 JavaScript 计算本身。

原因包括：

- `loc` 统计阶段需要读取大量 revision 文件内容
- 底层读取模型仍偏向“单文件一次 Git 调用”
- 大仓库会同时放大 sampled snapshot 数、文件数和 Git 子进程数

因此，在大仓库中，单纯增加并发度未必是最优解。很多情况下，“减少 Git 调用次数”比“同时发起更多 Git 调用”更有效。

## 5. 并行化维度分析

### 5.1 文件级并发

定义：

- 在单个 snapshot 内，把待统计的文件读取任务拆成多个并发工作项

当前状态：

- 已落地

收益：

- 改造成本低
- 不改变指标口径
- 对大多数中大型仓库都会带来直接收益

风险：

- 如果底层仍是单文件单次 Git 调用，继续增大并发很快会遇到收益递减
- 过高并发可能导致 Git 子进程过多、磁盘争用增大、系统负载升高
- 进度回调频率会随并发放大

结论：

- 文件级并发是当前的基础加速手段
- 但它不是最终的大头优化方向

### 5.2 模块级并发

定义：

- 在单个 snapshot 内，不同模块的 `loc` 统计分别并发执行

收益：

- 从概念上容易理解
- 对模块之间完全独立的场景有效

问题：

- 模块级并发和文件级并发本质上会竞争同一批底层 Git 读取资源
- 如果每个模块各自维护并发池，很容易形成多层并发叠加
- 同一文件是否唯一归属模块也需要继续保持严格约束

结论：

- 不建议把模块级并发作为独立主要优化层
- 更合理的做法是保留统一文件级任务池，让模块只作为聚合口径存在

### 5.3 Snapshot 级并发

定义：

- 不同 sampled snapshot 的模块识别、LOC 统计和 diff 读取并发执行

收益：

- 理论收益显著高于单 snapshot 内的文件级并发
- 对 `per-commit` 或高密度采样分析尤其有价值

主要难点：

- 当前实现依赖前一个 snapshot 来补齐“已消失模块”的零值点
- 进度模型从严格顺序推进变成多任务并发推进后会更复杂
- 同时处理多个 snapshot 会增加内存占用与 Git I/O 峰值

设计建议：

- 把“消失模块补零”从 snapshot 处理过程中下沉到统一聚合阶段
- 先让每个 snapshot 独立产出原始结果
- 最后按时间顺序做一次补齐和排序

结论：

- Snapshot 级并发是后续最值得投入的结构性加速点之一
- 但应建立在文件级并发和进度模型收口之后

### 5.4 语言 analyzer 级并发

定义：

- 在同一个 analysis task 中，`rust`、`node` 等 analyzer 并行执行

收益：

- 对混合仓库有直接收益
- 实现复杂度相对可控
- analyzer 结果本来就需要在末端汇总校验 snapshot 时间线

风险：

- 各 analyzer 会共享同一个仓库的 Git I/O 资源
- 若 analyzer 内部也继续扩张并发，容易形成资源争抢

结论：

- 这一层值得做
- 但必须配合统一的并发预算，而不是让每个 analyzer 自行放大并发度

### 5.5 Diff 归因阶段并发

定义：

- 对 `numstat` 结果做模块归因与聚合时使用并发处理

收益：

- 在 diff 行很多时有一定收益

限制：

- 这一阶段通常比文件内容读取更轻
- 如果只是对内存中的 `diffRows` 做简单归因和累加，CPU 不是主要瓶颈

结论：

- 不是优先优化方向
- 可在有明确基准证明其成为热点后再投入

### 5.6 任务级并发

定义：

- API 同时执行多个 analysis job

收益：

- 提升多仓库场景下的总体吞吐

风险：

- 如果缺少全局限流，多任务并发会互相争用 Git、磁盘和数据库
- 表面上是“更并行”，实际上可能让单任务尾延迟显著恶化

结论：

- 任务级并发需要调度，不应放任无上限并发
- 需要建立全局或按仓库的并发预算

## 6. 比增加并发更重要的优化

### 6.1 减少 Git 调用次数

当前最值得优先投入的优化，不是进一步堆高并发，而是减少 Git 调用次数。

可考虑的方向：

- 引入批量 blob 读取能力，例如基于 `git cat-file --batch`
- 为同一 revision 的文件读取建立缓存
- 尽量合并同一 snapshot 内的 Git 请求，减少进程启动开销

原因：

- 当前很多时间消耗在 Git 子进程启动与 I/O 往返
- “并发更多次单文件读取”通常不如“更少次批量读取”有效

### 6.2 进度上报节流

当前进度模型如果按每个文件、每条 diff row 都上报一次，会在并发度升高后放大两个问题：

- 高频回调本身会引入额外开销
- API 层若直接持久化每次进度更新，会让数据库写入成为新瓶颈

建议：

- 按时间窗口节流，例如 `200ms` 或 `500ms` 最多写一次
- 或按进度增量阈值节流，例如 percent 变化超过某个阈值才持久化
- 分离“内存态实时进度”和“持久化进度快照”

结论：

- 在继续增加并发层次之前，必须先收口进度写入频率

## 7. 推荐落地顺序

建议按以下顺序推进：

### Phase A: 稳定当前文件级并发

- 保持单 snapshot 内的受限文件级并发
- 把并发度收敛为可配置参数
- 建立基础性能基准，记录不同仓库下的耗时、CPU、内存与 I/O 表现

目标：

- 明确当前默认并发度下的稳定运行边界

### Phase B: 先做进度模型收口

- 为 snapshot 处理中的高频进度更新增加节流
- 避免每个微任务都触发持久化写入

目标：

- 防止进度系统反向吞掉并发收益

### Phase C: 减少 Git 调用次数

- 评估并接入批量 blob 读取方案
- 为同一 revision 的文件读取加入缓存或共享读取器

目标：

- 把主要瓶颈从“频繁启动 Git 子进程”转移出去

### Phase D: 引入语言 analyzer 级并发

- 在混合仓库中并发执行 `rust`、`node` 等 analyzer
- 使用统一并发预算控制底层 Git 读取资源

目标：

- 以较低复杂度获得混合仓库加速

### Phase E: 引入 Snapshot 级并发

- 让多个 sampled snapshot 并发产出原始结果
- 将模块消失补零与结果排序下沉到统一聚合阶段

目标：

- 获得结构性加速，尤其改善高密度采样场景

### Phase F: 建立任务级调度

- 为 API 后台分析任务增加全局限流或按仓库限流
- 区分单任务延迟优化与整体吞吐优化

目标：

- 避免多分析任务互相拖垮

## 8. 设计约束

后续所有并行化改造，都必须遵守以下约束：

- 并发必须是受限并发，不能无上限展开
- 结果顺序必须稳定，最终 `snapshots` 与 `points` 不能因并发漂移
- 同一文件在同一 snapshot 内的模块归属必须唯一
- 高并发下仍要支持中断，并保证中断后不会写入残缺结果
- 进度展示允许近实时，但不要求每个微任务都持久化
- 默认配置必须保守，优先保证稳定性而不是极限吞吐

## 9. 当前结论

当前已经落地的并行，本质上是：

- 单个 snapshot 内，按文件受限并发读取 revision 内容

当前尚未落地、但仍有明显收益空间的维度包括：

- 语言 analyzer 级并发
- Snapshot 级并发
- 任务级调度并发

但从整体收益排序看，后续最重要的优化并不是“继续增加并发层数”，而是：

1. 先节流进度上报与持久化
2. 再减少 Git 调用次数
3. 然后再把并行范围扩展到 analyzer 和 snapshot 层

因此，分析链路的下一阶段优化策略应当是：

- 先减少无效开销
- 再扩大并行范围
- 最后才继续抬高并发度

只有这样，才能避免系统从“分析慢”演变成“并发很多，但 Git、数据库和进度系统一起成为瓶颈”。
