# Code Dance 架构设计

> 版本：v0.3
> 更新时间：2026-04-09
> 变更摘要：补充 SQLite 实际落地 schema 文档，并校正总体设计中的存储表名

## 1. 项目定位

`code-dance` 是一个用于分析其他 Git 代码仓库历史演化的工具。本仓库本身不是被分析对象，而是分析器与展示平台。

目标是回答这类问题：

- 某个仓库从历史以来，各个代码模块的代码量如何变化
- 某个时间窗口内，各个模块的新增、删除、变更量如何分布
- 模块结构是否发生了重构、拆分、合并
- 哪些模块长期增长，哪些模块逐渐衰退

第一阶段重点指标是代码行数相关指标：

- `loc`: 某个时间点模块总代码行数
- `added`: 相邻采样点之间新增代码行数
- `deleted`: 相邻采样点之间删除代码行数
- `churn`: `added + deleted`

## 2. 设计目标

### 2.1 核心目标

- 面向任意 Git 仓库进行历史分析
- 支持按模块查看代码量时间序列
- 提供本地服务，通过前端页面交互式展示结果
- 从一开始就保证数据层与展示层解耦，以便后续支持 TUI

### 2.2 非目标

- 第一阶段不追求“每个 commit 都实时秒级分析”
- 第一阶段不做复杂的在线协作或多用户系统
- 第一阶段不强依赖某一种语言生态的语义分析

## 3. 总体原则

### 3.1 分层清晰

系统应拆成三层：

1. 分析层：读取 Git 历史，计算指标，落库
2. 服务层：向上提供统一的查询接口与任务接口
3. 展示层：Web 与 TUI 共享同一套数据接口，不直接接触 Git

### 3.2 模块定义可插拔

“模块”不能写死为某个目录结构。不同仓库可能需要不同定义方式，因此模块归类规则必须可配置、可扩展。

当前实现状态补充：

- Rust 仓库优先按 crate/workspace 识别模块。
- Node/Web 仓库优先按 `package.json` 的 workspace 配置识别模块。
- 对于没有 workspace 配置的 Node/Web 仓库，当前仍会回退到一套工程约定规则：
  - 某些容器目录按子目录拆分，例如 `apps/*`、`packages/*`、`services/*`、`os/*`
  - 其他顶层源码目录按目录拆分
- 这套 Node 回退规则是工程化启发式，不是最终形态，也不代表真正“自动理解”仓库模块边界。
- 后续应逐步演进为可配置规则，避免对目录命名约定产生过强依赖。

### 3.3 先做离线分析，再做交互查询

为了兼顾性能与复杂度，优先采用“分析任务 -> 存储结果 -> 前端查询”的模式，而不是每次页面刷新都直接扫描 Git 历史。

### 3.4 采样维度可并存

同一个仓库的分析结果不应只允许保留一种采样粒度。

当前实现状态补充：

- 分析任务的采样维度通过 `analysis_jobs.sampling` 显式记录。
- 同一个仓库可以并存多份 analysis，只要采样维度不同即可。
- 当前已支持的采样粒度包括：
  - `weekly`
  - `daily`
  - `per-commit`
- 同一个仓库、同一个采样维度下，默认只保留一份完成分析结果，避免重复记录同类数据。
- 前端工作台会按当前所选采样维度展示对应的分析结果，而不是把不同维度混在同一视图中。

## 4. 用户场景

### 4.1 Web 用户

- 选择待分析仓库路径
- 配置分支、采样频率、模块规则
- 启动分析任务
- 在页面查看模块 LOC 折线图、堆叠图、柱状图
- 按模块、时间范围、指标进行筛选

### 4.2 TUI 用户

- 在终端内启动分析任务
- 查看任务进度
- 浏览模块排行、时间序列摘要、变更热点

### 4.3 自动化用户

- 通过 CLI 或 API 批量分析多个仓库
- 导出结构化数据供后续处理

## 5. 总体架构

建议采用 monorepo 结构：

```text
apps/
  api/          # 本地服务，负责任务调度、查询接口、事件推送
  web/          # 前端页面，展示图表与筛选交互
  tui/          # 终端 UI，消费同一套服务接口

packages/
  domain/       # 核心领域模型
  contracts/    # API 类型、DTO、schema
  git/          # Git 读取与历史遍历
  analyzer/     # 指标计算、模块归类、采样逻辑
  storage/      # SQLite/缓存/数据访问层
  config/       # 仓库分析配置、模块规则解析
```

依赖方向应保持单向：

```text
web/tui -> api -> analyzer -> git
                 -> storage
contracts <-> api/web/tui
domain    <-> analyzer/storage
```

## 6. 核心模块设计

### 6.1 `packages/domain`

职责：

- 定义统一的领域对象
- 约束系统内的核心概念

建议包含：

- `RepositoryTarget`
- `AnalysisJob`
- `Snapshot`
- `ModuleDefinition`
- `MetricPoint`
- `SeriesQuery`

示例：

```ts
export type MetricPoint = {
  analysisId: string;
  ts: string;
  commit: string;
  module: string;
  loc: number;
  added: number;
  deleted: number;
  churn: number;
};
```

### 6.2 `packages/config`

职责：

- 定义分析配置格式
- 解析与校验模块规则

建议支持的模块定义方式：

1. `path-prefix`
2. `glob-map`
3. `manual-config`

配置示例：

```yaml
repository:
  branch: main
  sampling: weekly

modules:
  - name: compiler
    include: ["src/compiler/**"]
  - name: runtime
    include: ["src/runtime/**"]
  - name: tests
    include: ["tests/**"]
    exclude_from_loc: true
```

后续可扩展：

- Go package
- Rust crate
- Python module
- JavaScript workspace/package

### 6.3 `packages/git`

职责：

- 访问目标 Git 仓库
- 枚举主线历史或指定范围
- 读取某个提交下的文件树
- 读取 `numstat`、`diff`、重命名信息

建议提供的能力：

- `listCommits(target, options)`
- `readTree(commit)`
- `readNumstat(from, to)`
- `detectRename(from, to)`

这一层只关心 Git 事实，不关心“模块”或“图表”。

### 6.4 `packages/analyzer`

职责：

- 基于 Git 数据计算业务指标
- 执行采样策略
- 对文件归类到模块
- 生成时间序列结果

建议拆成几个分析器：

- `SnapshotAnalyzer`: 计算某个时间点各模块总 LOC
- `DeltaAnalyzer`: 计算相邻采样点之间的新增/删除/churn
- `ModuleClassifier`: 根据配置把文件映射到模块
- `SamplingPlanner`: 根据规则选出采样提交

当前落地限制：

- Rust analyzer 的模块边界主要来自 Cargo crate/workspace。
- Node analyzer 在有 workspace 配置时按 workspace package 建模。
- Node analyzer 在无 workspace 配置时，会使用目录约定回退来推断模块边界。
- 因此当前 Node 模块识别结果仍可能受仓库目录命名方式影响，需要在后续版本中补充显式配置能力或更稳定的推断机制。

### 6.5 `packages/storage`

职责：

- 保存分析任务与结果
- 提供查询接口
- 支持缓存与重跑判断

第一阶段建议使用 SQLite，原因：

- 本地工具部署简单
- 查询能力足够
- 同时适合 Web 和 TUI 共用

当前第一阶段实际落地的 SQLite 逻辑表：

- `repository_targets`
- `analysis_jobs`
- `analysis_progress`
- `snapshots`
- `analysis_modules`
- `module_metrics`

当前关于采样维度的补充说明：

- 现有表结构不需要为 `weekly`、`daily`、`per-commit` 单独拆表。
- 采样维度通过 `analysis_jobs.sampling` 区分，不同维度共享同一套 `snapshots`、`analysis_modules`、`module_metrics` 结构。
- 因此“是否需要重新分析”取决于仓库 ID 与采样维度组合，而不是仅由仓库 ID 决定。

详细字段说明与关系说明见 [2.storage.01-sqlite-schema.md](/home/zxr/work/github/code-dance/docs/design/2.storage.01-sqlite-schema.md)。

## 7. 数据模型

第一阶段的详细数据模型已经独立整理到 [2.storage.01-sqlite-schema.md](/home/zxr/work/github/code-dance/docs/design/2.storage.01-sqlite-schema.md)。

这里保留高层结论：

- `repository_targets` 记录被分析仓库
- `analysis_jobs` 记录分析任务
- `analysis_progress` 记录长任务进度
- `snapshots` 记录采样时间轴
- `analysis_modules` 记录某次分析识别出的模块集合
- `module_metrics` 记录每个采样点、每个模块的指标事实

这样的建模可以同时服务：

- 折线图：按 `module_key` 聚合 `loc`
- 柱状图：按时间点聚合 `added/deleted`
- 排行榜：按某个时间点排序 `loc` 或 `churn`

## 8. 分析流程

推荐的第一阶段流程如下：

1. 接收仓库路径、分支、模块配置、采样策略
2. 解析目标仓库并校验 Git 可访问性
3. 选出采样点
4. 对每个采样点读取文件树并统计模块 LOC
5. 基于相邻采样点生成 `added/deleted/churn`
6. 落库
7. 通过 API 暴露结果，并向前端推送任务进度

## 9. 采样策略

第一阶段不建议对“每个 commit”做全量快照，成本太高。建议优先支持：

- `daily`
- `weekly`
- `monthly`
- `tag-based`

默认推荐：

- 分支：`main` 或仓库默认分支
- 历史遍历：`first-parent`
- 采样频率：`weekly`

原因：

- 对大型仓库更稳定
- 更接近“主线演化”
- 图表可读性更好

## 10. 模块抽象与归类策略

“模块”不能被硬编码为目录。不同语言和生态中的模块边界并不一致：

- Rust 更接近 workspace / crate
- Go 更接近 module / package
- Python 更接近 package / top-level module
- JavaScript/TypeScript 更接近 workspace package / package.json 工程
- 一些通用仓库只能退回到目录规则

因此系统内部必须把“模块”设计成可插拔抽象，而不是单一规则。

### 10.1 两层模块概念

建议把模块拆成两层：

1. 技术模块：由语言生态或工程结构自动识别出的结构单元
2. 展示模块：面向用户展示的逻辑分组，可由一个或多个技术模块组成

示例：

- Rust 仓库中，一个 crate 是技术模块
- 前端页面上，多个 crate 可以合并展示为一个业务模块，如 `runtime`

这样设计的好处：

- 保留语言原生结构，便于分析准确性
- 允许用户按业务语义重新组织图表
- 避免不同语言仓库被硬塞进同一种目录模型

### 10.2 核心抽象

建议定义统一的模块单元与探测接口：

```ts
export type ModuleUnit = {
  key: string;
  name: string;
  kind: string;
  rootPath: string;
  files: string[];
  source: "auto" | "manual";
};

export interface ModuleProvider {
  kind: string;
  detect(ctx: DetectContext): Promise<ModuleUnit[]>;
}
```

其中：

- `key` 是稳定标识，例如 `rust:crate:parser`
- `name` 是展示名称
- `kind` 表示模块来源类型，例如 `rust-crate`、`go-package`、`path-prefix`
- `rootPath` 表示模块根路径
- `files` 表示某次快照下归属到该模块的文件集合
- `source` 区分自动识别与手工配置

系统最终需要解决的问题不是“模块长什么样”，而是：

**在某个 commit 下，如何为每个文件找到稳定的模块归属。**

### 10.3 Provider 机制

模块识别应通过 provider 链完成，而不是写死在分析器里。建议优先级如下：

1. `manual-config provider`
2. `language-aware provider`
3. `path-based fallback provider`

解释：

- 用户手工配置优先级最高，便于覆盖自动识别误差
- 语言感知 provider 负责识别 Rust crate、Go package 等原生结构
- 最后用路径规则兜底，保证任何仓库都至少能被分析

未命中的文件可按配置：

- 归入 `unknown`
- 忽略
- 单独列入 `unclassified`

### 10.4 技术模块与展示模块映射

建议支持第二层映射，把多个技术模块合并为一个展示分组。

配置示例：

```yaml
module_groups:
  - name: runtime
    members:
      - rust:crate:vm
      - rust:crate:executor
      - rust:crate:scheduler
```

这样可以同时支持两种视图：

- 技术视图：查看原生模块演化
- 业务视图：查看逻辑分组演化

### 10.5 多语言实现策略

不建议一开始做统一 AST 或语义级分析，成本过高。第一阶段优先做“基于工程清单文件和目录结构的模块识别”。

建议支持路线：

1. `manual-config`
2. `path-prefix`
3. Rust workspace / crate
4. Node.js workspace / package
5. Go module / package
6. Python package

这条路线的特点是：

- 足够稳健
- 对大多数工程已有结构兼容性好
- 不依赖复杂语法分析
- 可以渐进扩展

### 10.6 Rust 优先支持方案

Rust 是第一批优先支持的语言，原因：

- Rust 仓库普遍具有明确的 workspace / crate 边界
- `Cargo.toml` 结构稳定，适合做模块探测
- crate 作为技术模块，天然适合做时间序列分析

Rust provider 的基本策略：

1. 在目标 commit 下识别根 `Cargo.toml`
2. 解析 workspace members
3. 为每个 crate 建立 `ModuleUnit`
4. 将 crate 根目录下的文件归属到对应模块
5. 未归属文件再交给 fallback provider

Rust 第一阶段的模块粒度建议是：

- 默认按 crate 统计
- 可选把 workspace 根下的非 crate 文件归入 `workspace-root`
- 后续再考虑更细的 `module tree` 或 `src/bin/*` 粒度

Rust 相关命名建议：

- `kind = rust-workspace`
- `kind = rust-crate`

### 10.7 设计注意点

- 支持文件排除规则，如 `vendor/`、`dist/`、生成代码
- 支持测试目录是否纳入 LOC
- 支持未命中文件归为 `unknown` 或直接忽略
- 模块标识要尽量稳定，避免因展示名变化导致历史曲线断裂
- 同一个文件在一次快照内只能归属于一个技术模块

## 11. LOC 统计策略

第一阶段建议定义清楚“代码行数”：

- 默认不区分语言，只做文本行统计
- 默认排除二进制文件
- 是否排除空行、注释行应可配置

建议分两层：

1. 粗粒度 LOC：统计非空文本行
2. 语言感知 LOC：后续按语言扩展注释识别

MVP 先实现第一层即可。

## 12. API 设计

### 12.1 任务接口

- `POST /api/analyses`
- `GET /api/analyses/:id`
- `GET /api/analyses`

### 12.2 查询接口

- `GET /api/modules?analysisId=...`
- `GET /api/series?analysisId=...&metric=loc`
- `GET /api/distribution?analysisId=...&snapshotId=...`
- `GET /api/ranking?analysisId=...&metric=churn`

### 12.3 事件接口

- `GET /api/events/:analysisId`

事件接口用于向 Web/TUI 推送：

- 任务开始
- 当前采样进度
- 分析完成
- 分析失败

## 13. Web 展示层

Web 负责展示与交互，不负责计算。建议页面包含：

- 仓库输入与配置表单
- 分析任务列表
- 模块 LOC 折线图
- 模块堆叠面积图
- 时间窗口柱状图
- 模块筛选器
- 时间范围选择器

图表能力建议：

- 多模块折线对比
- Hover 查看某时间点详细值
- 开关 `loc/added/deleted/churn`
- 支持隐藏噪音模块
- 图表库统一使用 ECharts

## 14. TUI 展示层

TUI 与 Web 共享相同 API 和数据契约，重点做高密度信息浏览：

- 任务状态面板
- 模块排行
- 时间序列摘要
- 简化版柱状图或 sparkline

TUI 不应重新实现分析逻辑，只消费服务层数据。

## 15. 技术选型建议

为了降低实现成本，第一阶段建议统一用 TypeScript：

- `api`: Fastify
- `web`: React + Vite + ECharts
- `tui`: Ink 或 blessed
- `storage`: SQLite
- `contracts`: zod + TypeScript types

统一技术栈的好处：

- 数据模型共享简单
- Web/TUI/服务端契约一致
- 便于快速迭代 MVP

## 16. 性能与缓存

大仓库分析时，性能会成为关键问题。第一阶段就应预留以下能力：

- 采样而不是逐 commit 全量回放
- 结果落库，避免重复分析
- 按分析配置计算缓存 key
- 支持断点续跑

后续可扩展：

- 基于 `git log --numstat` 的增量模式
- 快照缓存
- 并发统计文件 LOC

## 17. 风险点

### 17.1 模块重命名

目录迁移或模块拆分会导致曲线断裂。第一阶段可先接受这一限制，后续增加“模块映射迁移规则”。

### 17.2 LOC 定义不统一

不同语言、空行、注释行的处理会影响结果。因此必须把统计口径明确写入配置与页面说明。

### 17.3 大仓库性能

如果仓库体量很大，按周全量快照也可能耗时较长，因此任务化与进度反馈是必须的。

## 18. MVP 范围

MVP 目标不是覆盖所有场景，而是跑通主链路：

1. 输入一个本地 Git 仓库路径
2. 配置分支与模块规则
3. 按周采样主线历史
4. 统计每个模块的 LOC 与增删变化
5. 通过本地服务提供查询接口
6. 在 Web 页面展示折线图与柱状图

MVP 首批模块识别能力建议明确为：

- 手工配置模块
- 目录前缀模块
- Rust workspace / crate 模块

TUI 在 MVP 可以先支持：

- 查看任务列表
- 查看模块排行
- 查看简化版时间序列摘要

- 基础图形化展示

## 20. 当前结论

本仓库应被定义为“Git 仓库历史模块演化分析工具”，而不是某个单独的图表应用。

第一阶段最重要的不是 UI，而是保证以下三件事设计正确：

1. 模块定义可配置
2. 分析结果可持久化与复用
3. Web 与 TUI 消费同一套服务接口

如果这三点成立，后续无论增加更多指标、更多图表还是更多终端，成本都会明显降低。
