# 后端服务性能优化设计

> 版本：v0.1
> 更新时间：2026-04-10
> 变更摘要：新增后端服务性能专项设计，整理当前卡顿现象、证据链、瓶颈分类与推荐优化顺序

## 1. 背景

最近在真实使用中，`code-dance` Web 端出现了明显的“加载数据很久”“有时页面像卡死一样”的问题。

静态代码审查与本地运行日志都表明，这不是单一慢 SQL 或单一前端 bug，而是以下问题叠加：

- 前端高频轮询命中了过重接口
- 分析详情页首屏会并发触发一组重查询
- 存储层使用同步 `better-sqlite3`，重查询会阻塞 API 进程
- 分析任务进行中会持续写 SQLite，与查询请求争用同一进程
- 某些接口采用“全量读取 + JS 聚合/裁剪”的实现方式，放大了内存和序列化压力

其中最严重的问题已经在实际日志中暴露为 `/api/analyses/:id` 长时间无响应，最终在序列化阶段抛出：

- `RangeError: Invalid string length`

这说明当前“完整分析结果接口”在大结果集场景下已经不只是慢，而是可能直接不可用。

## 2. 目标

- 梳理当前后端卡顿的主要原因与证据链
- 区分“接口设计问题”“查询实现问题”“存储竞争问题”“前端触发问题”
- 明确哪些问题是 P0，哪些可以后续迭代
- 给出一条可落地的优化顺序，优先解决最影响体感的问题

## 3. 非目标

- 本文不讨论前端视觉改版
- 本文不引入分布式调度、远程数据库或多机执行
- 本文不重新定义分析指标口径
- 本文不覆盖 analyzer 语义正确性问题

## 4. 当前现象与证据

### 4.1 运行日志中的直接现象

本地服务日志显示：

- `/api/repositories` 响应时间约 `0.4ms - 5ms`
- `/api/analysis-summaries` 响应时间约 `28ms - 30ms`
- `/api/analyses/:id` 响应时间约 `24s - 26s`
- `/api/analyses/:id` 最终返回 `500`
- 错误发生在 Fastify `JSON.stringify` 阶段，并抛出 `RangeError: Invalid string length`

这说明当前最重的热点不是首页基础元数据，而是单个分析详情的完整结果加载。

## 5. 问题分类

### 5.1 P0: 轮询命中了过重接口

当前运行中分析的进度刷新，并不是读取轻量进度，而是反复读取完整分析结果。

链路如下：

1. 首页定时器每秒遍历运行中任务
2. 对每个任务调用 `/api/analyses/:id`
3. 后端读取整份分析结果
4. Fastify 再尝试序列化整份大对象

问题在于：

- 进度轮询不需要 `points` 和 `candles`
- 完整结果对象随着采样点和模块数增长而迅速膨胀
- 高频重复读取和序列化会直接拖慢整个 API 进程
- 在极端情况下，序列化会失败并返回 `500`

这是当前最需要优先修掉的问题。

### 5.2 P0: 详情页首屏请求风暴

分析详情页虽然 UI 上已经是单图聚焦，但数据层仍然采用“先把整套图表数据全部拉齐”的策略。

当前首屏会并发请求：

- `GET /api/modules`
- `GET /api/series` x 4
- `GET /api/candles`
- `GET /api/distribution` x 4

此外，如果外层状态尚未加载完成，还可能额外触发：

- `GET /api/analyses/:id`
- `GET /api/repositories`
- `GET /api/analysis-summaries`

风险包括：

- 同一分析会在短时间内被重复读取多次
- 多个重查询集中压到同一个同步 SQLite 连接上
- 首屏就拉取用户不一定会看的图表数据
- 请求失败或变慢时，用户体感会直接退化成“页面很卡”

### 5.3 P0: `/api/analyses/:id` 返回对象过大

完整分析结果接口当前承担了两种角色：

- 运行中任务的进度刷新
- 已完成分析的重结果读取

这两个角色的数据需求完全不同，但当前复用了同一个大接口。

后果：

- 运行中任务为了拿进度也要承担重载荷
- 已完成分析在大仓库下会产生极大 JSON
- 即便后端读库成功，也可能在序列化阶段失败

这意味着该接口必须拆分，而不是继续沿用。

### 5.4 P1: `series/candles/distribution/ranking` 查询默认走全量读

当前查询实现有明显的“先全量拿，再内存整理”的特征：

- `querySeries()` 读出 analysis 下所有匹配行，再在 JS 中按模块和 timeline 重组，见 [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L476)
- `queryCandles()` 同理，见 [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L514)
- `queryRanking()` 先走完整 distribution，再 `slice(limit)`，见 [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L593)

这会带来：

- 大结果集搬运
- 大量对象分配
- 不必要的排序和裁剪
- 网络响应体偏大

其中 `ranking` 的实现尤其浪费，因为用户只要 Top N，但后端每次都要先算全量。

### 5.5 P1: 首页 summary 查询存在 N+1

`listAnalysisSummaries()` 会：

1. 对 `analysis_jobs + analysis_progress + snapshots` 做聚合
2. 再对每条 analysis 单独调用 `resolveSnapshot()` 查最新 snapshot

实现见：

- [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L321)
- [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L375)

这属于典型 N+1。

此外，新建分析前还会：

- 先全量读取 summary 列表
- 再在内存中 `.find()` 判断是否已有完成分析

见 [server.ts](/home/zxr/work/github/code-dance/apps/api/src/server.ts#L350)。

随着历史分析记录增多，这两条路径都会逐渐恶化。

### 5.6 P1: 分析过程的同步写库会加剧卡顿

分析任务在运行过程中会持续更新：

- `analysis_jobs.status`
- `analysis_progress`

而 analyzer 内部在文件处理和 diff 归因过程中都会多次尝试发布进度。

虽然当前有节流，但问题仍然存在：

- `better-sqlite3` 是同步写
- 分析和查询共享同一进程
- 大量写入会和前台读请求相互挤压

这会导致：

- 正在分析时，页面更容易感觉卡
- 某些读请求延迟明显抬高

### 5.7 P2: 模块探测接口是现算而非缓存

`/api/repositories/:id/modules` 当前不是轻量查询，而是重新跑模块识别。

入口见：

- [server.ts](/home/zxr/work/github/code-dance/apps/api/src/server.ts#L169)

这条路径的问题在于：

- 大仓库会触发递归文件扫描
- 混合仓库可能让多个 provider 各自扫一遍
- 打开模块页时，用户会把这类等待理解成“数据接口很慢”

这不是当前最致命的问题，但确实是一个独立性能热点。

### 5.8 P2: 索引没有对准主要查询路径

当前主要索引为：

- `idx_module_metrics_analysis_ts`
- `idx_module_candles_analysis_ts`

定义见 [index.ts](/home/zxr/work/github/code-dance/packages/storage/src/index.ts#L821)。

但主要查询更接近：

- `where analysis_id = ? order by module_key, snapshot_seq`
- `where analysis_id = ? and snapshot_seq = ? order by value desc`
- `where analysis_id = ? and module_key in (...) order by snapshot_seq`

这意味着当前索引和热点 SQL 并不对齐，容易导致：

- 额外排序
- 低效扫描
- 热点接口放大延迟

这里的判断基于源码审查，还没有结合 `EXPLAIN QUERY PLAN` 做进一步验证。

## 6. 根因总结

当前后端“偶发很卡”的本质，不像是某一条 SQL 单独慢，而是三个结构性问题叠加：

### 6.1 接口分层不清

把“轻量进度读取”和“完整结果读取”混进了同一接口。

### 6.2 查询策略偏全量

多个接口采用“全量读取 + JS 聚合/裁剪”的实现，缺少 SQL 层裁剪和惰性加载。

### 6.3 同步 SQLite 与重任务共进程

分析任务、轮询请求、详情页重查询都挤在同一个 Node 进程和同一个同步 SQLite 连接上。

## 7. 优化目标

围绕当前问题，优化目标应该按以下顺序定义：

### 7.1 第一阶段目标

- 消除 `/api/analyses/:id` 在轮询场景下的错误使用
- 避免大结果对象反复读取和序列化
- 让运行中任务的页面不再出现 20 秒以上卡顿或 500

### 7.2 第二阶段目标

- 降低详情页首屏请求数
- 避免一次进入详情页就把所有图表数据查完
- 让图表加载从“请求风暴”改为“按需读取”

### 7.3 第三阶段目标

- 收紧重查询的结果规模
- 补齐与查询路径匹配的索引
- 减少无意义重读和 N+1

### 7.4 第四阶段目标

- 缓解分析过程中的读写争用
- 减轻模块探测和辅助查询的重复成本

## 8. 推荐改造方案

### 8.1 P0: 拆分轻量 progress/summary 接口

新增轻量接口，只返回：

- `job`
- `progress`
- `snapshotCount`
- `latestSnapshot`

适用场景：

- 首页轮询
- 运行中分析的刷新
- 任务列表状态展示

同时：

- 停止使用 `/api/analyses/:id` 做运行中任务轮询
- 把前端轮询切到新接口

这样可以直接切断当前最严重的 25 秒慢请求链路。

### 8.2 P0: 把详情页改成按需加载

详情页不再在首屏并发拉全套图表数据。

建议改成：

- 首屏只请求最基础的 summary 和当前默认图所需数据
- 切换图表 tab 时再加载对应数据
- 已加载数据做页面级缓存

最直接的收益：

- 首屏请求数骤降
- 同一分析的重复查询减少
- 慢接口不会在首次打开时全部叠加

### 8.3 P0: 收紧 `/api/analyses/:id` 的职责

该接口不应继续承担高频刷新用途。

建议：

- 明确它只用于“重结果读取”
- 后续进一步拆成：
  - summary 接口
  - timeline 接口
  - series/candles/distribution 专项接口

必要时还可以考虑：

- 限制该接口返回规模
- 在超大结果场景直接拒绝整包返回

### 8.4 P1: 让 `ranking` 真正使用 SQL `limit`

`queryRanking()` 应直接在 SQL 层完成：

- 按 metric 排序
- 限制返回前 N 条

而不是先取完整 distribution 再 `slice()`。

### 8.5 P1: 避免 `series/candles` 默认全量返回

建议后续收紧接口策略：

- 默认不返回所有模块
- 优先支持显式 `moduleKeys`
- 或者只返回 Top N / 聚合视图

这样可以显著降低：

- SQL 结果集大小
- JS 组装成本
- 网络响应体大小

### 8.6 P1: 去掉首页 summary 的 N+1

`listAnalysisSummaries()` 应一次查询直接拿到：

- snapshotCount
- latestSnapshot

避免每条 analysis 再补一次 `resolveSnapshot()`。

同时，新建分析前的查重应改成定向查询，而不是：

- 全量读 summary
- 再内存 `.find()`

### 8.7 P1: 降低分析过程写库频率

建议方向：

- 合并部分重复写入
- 降低 `updateJob + upsertProgress` 的重复调用
- 在 `status` 未变化时避免重复写 `analysis_jobs`

目标不是取消进度更新，而是避免高频同步写。

### 8.8 P2: 为模块探测增加缓存

建议按以下维度缓存模块探测结果：

- `repositoryId`
- `HEAD commit`
- `detectedKinds`

这样至少可以避免：

- 同一个仓库在短时间内反复打开模块页时重复扫描

### 8.9 P2: 重做索引

建议围绕真实热点路径补索引，例如：

- `(analysis_id, snapshot_seq, module_key)`
- `(analysis_id, module_key, snapshot_seq)`
- 适合 distribution/ranking 的 `(analysis_id, snapshot_seq, loc)` 等变体

具体索引组合需要结合：

- 实际数据规模
- SQLite `EXPLAIN QUERY PLAN`
- 真实最慢端点的 profiling 结果

## 9. 推荐实施顺序

### 9.1 第一批

- 新增轻量 progress/summary 接口
- 首页轮询切到轻量接口
- 停止对运行中任务使用 `/api/analyses/:id`

这是收益最高、风险最可控的一批。

### 9.2 第二批

- 详情页改成按需加载
- 避免首屏并发全套图表查询
- 去掉排行图的重复独立请求

### 9.3 第三批

- 优化 `ranking`
- 去掉 `analysis-summaries` 的 N+1
- 把查重改为定向查询
- 降低写库频率

### 9.4 第四批

- 为模块探测加缓存
- 基于真实慢查询补索引
- 视结果决定是否进一步拆分存储或执行模型

## 10. 验证方式

优化后建议至少验证以下几类场景：

### 10.1 运行中分析

- 有分析任务在跑时，首页不再持续出现长时间 pending
- 轮询请求应稳定落在轻量接口
- 不再出现 `/api/analyses/:id` 每秒触发

### 10.2 大结果分析详情页

- 打开详情页时，首屏请求数明显下降
- 不再出现 20 秒以上的同步等待
- 默认图表可在可接受时间内完成展示

### 10.3 大结果完整接口

- `/api/analyses/:id` 在大结果下不再触发 `Invalid string length`
- 如继续保留整包返回，应明确结果规模是否受控

### 10.4 数据量增长场景

- 分析记录变多后，`/api/analysis-summaries` 仍保持稳定
- 新建分析前的查重不再和历史总量线性恶化

## 11. 当前结论

当前后端性能问题已经不是“是不是该优化”，而是接口职责和查询策略需要收口。

最核心的判断是：

- 现在最危险的不是分析任务本身慢
- 而是前端把“完整分析结果接口”当成了高频状态接口来用

因此第一优先级必须是：

- 拆轻量接口
- 切换轮询路径
- 避免继续把重结果整包反复读取和序列化

在此基础上，再推进详情页按需加载、查询裁剪、N+1 清理、缓存与索引优化，整体体感才会真正稳定下来。
