# Analyzer 性能与并发设计

## 目标

- 说明当前 analyzer 的主要性能瓶颈。
- 明确哪些阶段适合并发，哪些阶段不应过早复杂化。
- 给出可逐步落地的优化顺序，避免一次性引入过多机制。

## 当前瓶颈

### 执行模型

- 当前 analyzer 的主流程仍然是串行的：按 sampled commit 逐个分析 snapshot。
- 每个 snapshot 内部也基本是串行的：先识别模块，再逐模块统计 LOC，再逐条处理 diff stat。
- LOC 统计阶段目前按文件逐个读取 revision 内容，属于大量细粒度 Git I/O。

### 为什么大仓库会慢

- 大仓库通常同时具备更多 sampled commit、更多模块和更多文件。
- 当前实现会对大量文件逐个执行 revision 读取，导致 Git 子进程调用次数很多。
- 因此主要瓶颈更偏向 Git I/O 和进程调度，而不是 JavaScript 计算本身。

## 适合引入并发的层次

### 1. 文件级并发

- 在单个 snapshot 内，模块 LOC 统计可以改为限流并发，而不是逐文件串行读取。
- 推荐做法是引入统一的并发池，例如 `4` 到 `8` 个并发文件读取任务。
- 这类改造通常收益最高，且不改变统计口径。
- 进度上报不能再依赖严格顺序，应改为基于共享计数器累计 `processedFiles`。

### 2. 模块级并发

- 单个 snapshot 下，不同模块的 LOC 汇总也可以并发执行。
- 但模块之间可能共享底层 Git 读取配额，因此更推荐复用统一文件级并发池，而不是每个模块各自再开一层并发。
- 模块级并发应建立在文件级限流之上，避免并发层层叠加导致 Git 子进程过载。

### 3. Snapshot 级并发

- 不同 sampled commit 的模块识别、LOC 统计、diff 读取，本质上大多可以独立计算。
- 当前实现中唯一明显偏串行的部分，是需要参考前一个 snapshot 来补齐“已消失模块”的零值点。
- 这一依赖可以下沉到聚合后处理阶段：先并发产出每个 snapshot 的原始分析结果，再统一补齐缺失模块点位。
- Snapshot 级并发收益更高，但实现复杂度、内存占用和进度统计复杂度也更高，应放在文件级并发之后。

## 不建议优先投入的方向

### 不建议第一步就引入 `worker_threads`

- 当前瓶颈更偏向 Git 子进程和磁盘 I/O，而不是 JavaScript CPU 计算。
- 如果底层仍然保持“单文件一次 Git 调用”的模型，单纯把任务转移到 worker 上，收益通常有限。
- `worker_threads` 更适合 CPU 密集型场景，不适合作为当前阶段的第一优化手段。

## 比并发更重要的优化点

### 减少 Git 调用次数

- 当前最重的开销之一，是按文件逐个执行 revision 读取。
- 在大仓库中，真正的大头优化往往是减少 Git 调用次数，而不仅仅是增加并发数。

### 可考虑的方向

- 引入批量 blob 读取能力，例如基于 `git cat-file --batch`。
- 为同一 revision 的文件读取建立缓存。
- 尽量合并同一 snapshot 下的 Git 请求，减少进程启动和 I/O 往返。

## 推荐落地顺序

1. 抽出共享的“限流并发 LOC 计数器”，供 `rust` 和 `node` analyzer 复用。
2. 保持 snapshot 外层串行，只在单个 snapshot 内做文件级并发。
3. 调整进度上报模型，使其适配乱序完成的任务。
4. 补充大仓库性能基准，验证并发度上升后的收益与资源占用。
5. 在确认收益稳定后，再评估 snapshot 级并发。
6. 最后再考虑批量 Git 读取，把“并发更多次调用”升级为“更少次调用”。

## 实现约束

- 并发必须是“限流并发”，不能无上限展开。
- 并发度应允许配置，并提供保守默认值。
- 结果顺序必须保持稳定，最终输出的 `snapshots` 与 `points` 顺序不能因为并发而漂移。
- 同一文件在同一 snapshot 内的模块归属仍必须唯一，不能因为并发拆分而引入重复统计。
- 进度回调应允许高频更新，但需要避免为每个微小任务都触发重型持久化。
